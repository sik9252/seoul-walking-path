# 서울 전용 산책 및 걷기 앱 PRD (MVP)

## 문서 정보
- 제품 가칭: **서울발자국**
- 대상 플랫폼: **iOS / Android**
- 문서 버전: **1.0**
- 작성 기준일: **2026-02-17**

## 핵심 가치(1문장)
서울에서 산책 코스를 쉽게 찾고, 걷는 동안 GPS로 트래킹하며 코스 정보와 주변 시설을 실시간으로 확인한다.

## MVP 핵심 기능
- 서울 내 산책 코스 정보 제공(총 길이, 예상 소요시간, 난이도, 테마 태그 등)
- GPS 기반 산책 트래킹(시작/일시정지/종료, 이동 경로 및 통계 기록)
- 산책 중 지도 화면에서 코스 라인과 내 위치, 주변 시설(예: 화장실/편의점/역)을 표시
- 산책 종료 후 요약 및 기록 저장(최근 기록 조회)

# 1. 문서 개요
본 문서는 서울 전용 산책 및 걷기 앱의 MVP(최소 기능 제품) 요구사항을 정의한다.  
MVP 목표는 **‘코스 탐색 → 산책 시작 → 산책 중 안내 → 산책 종료 기록’**의 핵심 플로우를 안정적으로 제공하는 것이다.

| 항목 | 내용 |
|---|---|
| 제품 범위 | 서울(행정구역 기준) 내 산책/걷기 코스 |
| 플랫폼 | iOS / Android |
| 핵심 사용자 가치 | 서울 코스 정보 + GPS 트래킹 + 주변 시설(POI) 실시간 확인 |
| MVP 성공 기준(요약) | 트래킹 안정성, 코스 탐색-시작 전환율, 산책 종료 저장률 |

---

# 2. 배경과 문제 정의
사용자는 ‘서울에서 걷기 좋은 코스’를 찾을 때 지도 앱/블로그/공식 앱/만보기 앱 등을 혼합해 사용한다.  
하지만 산책 목적에 최적화된 경험(코스 큐레이션 + 산책 중 맥락 정보 + 기록)이 단절되어 있다.

## 사용자 핵심 Pain Points
- 코스 정보가 분산되어 있고, 거리/소요시간/난이도 기준이 제각각이라 비교가 어렵다.
- 지도 앱은 길찾기 중심이라 ‘산책 경험’(테마, 하이라이트, 산책로 특성, 편의시설)을 한 번에 파악하기 어렵다.
- 산책 중에는 ‘지금 코스 위에 있는지’와 ‘근처 화장실/편의점/역’ 같은 정보가 즉시 필요하다.

**제품 기회:** 서울 전용으로 범위를 좁혀 코스 품질과 안내 UX를 높이고, 산책 중 ‘지금 여기’ 중심 정보를 빠르게 제공한다.

---

# 3. 목표와 비목표

## Goals (MVP에서 반드시 달성)
1. 사용자가 내 주변 또는 특정 지역에서 산책 코스를 탐색하고 비교할 수 있다.
2. 코스 상세에서 거리/예상시간/난이도/지도 프리뷰를 확인하고 산책을 시작할 수 있다.
3. GPS 기반으로 산책 경로와 통계(시간/거리)를 안정적으로 기록한다.
4. 산책 중 코스 라인과 주변 시설(POI)을 지도에서 확인할 수 있다.
5. 산책 종료 후 요약을 보고 기록을 저장/조회할 수 있다.

## Non-goals (MVP에서 제외 또는 후순위)
- 현금성 포인트/리워드, 기부 연동, 대규모 챌린지/리그
- 커뮤니티 피드/팔로우, 리뷰 시스템(단, 간단한 ‘제보’는 후순위로 고려)
- 사용자 커스텀 코스 제작/GPX 업로드
- 고도화된 건강 데이터 분석/코칭

---

# 4. 타깃 사용자
- **서울 거주 직장인:** 평일 저녁/주말에 30–90분 산책. 접근성(지하철)과 편의시설을 중시.
- **가벼운 걷기 선호 사용자:** 무리 없는 난이도, 휴식 포인트(벤치/카페), 안전한 루트를 선호.
- **서울 여행자/나들이 사용자:** 테마(한강/공원/성곽/야경), 사진 포인트, 길을 잃지 않는 안내를 원함.

---

# 5. 핵심 사용자 시나리오

## 시나리오 A: 내 주변 코스 찾기 → 바로 걷기
1. 앱 실행 후 위치 권한을 허용한다.
2. 홈에서 ‘내 주변 추천 코스’를 확인한다.
3. 코스 상세(거리/시간/지도)를 보고 ‘산책 시작’을 누른다.
4. 트래킹 화면에서 코스와 내 위치를 보며 걷는다.
5. 필요 시 주변 화장실/편의점/역을 확인한다.
6. 종료 후 요약을 저장하고 기록에서 다시 확인한다.

## 시나리오 B: 특정 지역(예: 여의도/성수) 탐색
1. 검색/필터로 특정 지역 코스를 조회한다.
2. 거리/시간/테마로 후보를 비교한다.
3. 선택한 코스의 시작점으로 이동(외부 지도 딥링크) 후 산책을 시작한다.

---

# 6. MVP 범위
MVP에서는 코스 탐색과 산책 중 안내의 핵심 경험을 우선한다.  
코스 수는 운영 가능한 범위(예: **30–80개**)에서 시작한다.

| 포함(Include) | 제외/후순위(Exclude) |
|---|---|
| 코스 탐색(추천/검색/필터/정렬) | 현금성 보상/앱테크, 기부 |
| 코스 상세(거리/시간/난이도/지도 프리뷰) | 대규모 챌린지/리그 |
| GPS 트래킹(시작/일시정지/종료) | 커뮤니티/팔로우/피드 |
| 산책 중 지도(코스/내 위치/주변시설 POI) | 커스텀 코스 제작/GPX 업로드 |
| 산책 종료 요약 및 기록 저장/조회 | 고도화 개인화/건강 코칭 |
| 즐겨찾기(코스 저장) | 고급 오프라인 지도(부분 캐시부터 고려) |

# 7. 정보 구조 및 주요 화면

## 탭 구조(권장)
- 홈
- 코스
- 기록
- 마이(설정)

## 주요 화면 목록
- 온보딩/권한 요청(위치, 백그라운드 위치, 알림(선택))
- 홈: 내 주변 추천, 테마 추천, 빠른 필터
- 코스 목록: 검색/필터/정렬
- 코스 상세: 핵심 메타 + 지도 프리뷰 + 주변 시설 요약
- 산책 트래킹: 지도 중심(코스 라인/내 위치/트랙/POI)
- 산책 종료 요약: 지도 + 통계 + 저장
- 기록 목록/기록 상세
- 설정(단위, 개인정보, 배터리/정확도 모드)

# 8. 기능 요구사항
우선순위는 **P0(필수), P1(중요), P2(후순위)**로 구분한다. 각 항목은 수용 기준(Acceptance Criteria)을 포함한다.

## FR-1. 코스 데이터/목록 제공 (P0)
서울 내 산책 코스를 리스트로 제공하고 검색/필터/정렬을 지원한다.

- 코스 메타: 코스명, 지역(구/동), 총거리(km), 예상 소요시간(분), 난이도(쉬움/보통/어려움), 테마 태그(한강/공원/숲길/야경 등)
- 정렬: 추천순(기본), 거리 짧은순, 내 위치 가까운순
- 필터: 거리(1–3 / 3–6 / 6+), 소요시간(30 / 60 / 90+), 지역(구), 테마

**수용 기준**
- 홈/코스탭에서 코스 리스트가 1초 내 로딩(캐시 포함 시).
- 필터 적용 시 결과가 즉시 반영되며 ‘필터 초기화’를 제공.

## FR-2. 코스 상세 화면 (P0)
코스를 시작하기 전 필요한 정보를 한 화면에서 확인한다.

- 표시: 거리, 예상 시간, 난이도, 루프 여부(원점회귀), 길 유형(포장/흙길/계단 등)
- 지도 프리뷰: 코스 라인(폴리라인) 표시
- 시작점/종료점: 주소 또는 랜드마크
- 주변 시설 요약: 카테고리별 개수(예: 화장실 3, 편의점 2 등)
- 액션: 산책 시작, 즐겨찾기, 시작점 길찾기(외부 지도 딥링크)

**수용 기준**
- 프리뷰 지도에서 ‘내 현재 위치’와 ‘코스 시작점’을 확인 가능.
- ‘산책 시작’ 시 트래킹 화면으로 전환되고 코스 폴리라인이 표시됨.

## FR-3. GPS 트래킹(산책 세션) (P0)
산책 시작-종료까지 이동 경로를 기록하고 통계를 산출한다.

- 세션 제어: 시작/일시정지/재개/종료
- 기록: 시작/종료 시각, 총 시간, 총 거리, 이동 경로(좌표/폴리라인), 평균 속도 또는 페이스(선택)
- 노이즈 방지: 비정상 점프 필터, 정지 상태 흔들림 누적 방지(스무딩)
- 백그라운드 기록: 화면 꺼짐/백그라운드에서도 가능한 범위에서 지속(권한/배터리 안내 포함)

**수용 기준**
- 테스트 코스(도심/공원 혼합)에서 30분 산책 후 거리/시간 오차가 사용자 경험을 해치지 않는 수준으로 유지(팀이 기준 수치 정의).
- 백그라운드에서 기록 중단이 발생할 경우 사용자에게 즉시 고지(배너/토스트)하고 복구 경로 안내.

## FR-4. 산책 중 지도 UI: 코스 + 주변 시설(POI) 표시 (P0)
산책 중 ‘지금 어디고, 주변에 뭐가 있는지’를 즉시 확인할 수 있어야 한다.

**POI 카테고리(MVP 권장 5종)**
- 화장실
- 편의점
- 카페
- 지하철역
- 버스정류장 또는 공영주차장(택1)

- 지도 표시: 내 현재 위치(방향 포함), 선택 코스 라인, 실제 이동 트랙, POI 아이콘
- POI 로딩: 내 위치 반경 또는 코스 라인 주변 버퍼 기준. 과다 표시 시 클러스터링.
- POI 탭 시 정보 카드: 이름, 현재 위치 기준 거리, 길찾기(외부 지도) 버튼

**수용 기준**
- 트래킹 화면 진입 후 2초 내 지도와 내 위치가 표시됨.
- POI 탭 시 상세 카드가 열리고 ‘길찾기’ 액션이 동작함.

## FR-5. 산책 종료 요약 & 기록 (P0)
산책 후 요약을 제공하고 기록을 저장/조회할 수 있어야 한다.

- 종료 요약: 총 거리, 총 시간, 평균 페이스(선택), 지도(트랙/코스 표시)
- 기록 저장: 기기 로컬 저장(필수). 계정/서버 동기화는 선택(후순위).
- 기록 리스트: 날짜/코스명/거리/시간 요약 카드

**수용 기준**
- 앱 재실행 후에도 기록이 유지되고, 기록 상세에서 지도 트랙이 재표시됨(로컬 기준).

## FR-6. 즐겨찾기/저장 (P1)
- 코스 상세에서 즐겨찾기 토글(하트).
- 즐겨찾기 목록에서 빠르게 접근.

## FR-7. 이슈 제보(간단) (P2)
- ‘길이 막혔어요/정보가 달라요/시설이 없어요’ 등 선택 + 텍스트.
- 운영자 확인을 위한 최소 데이터 수집(코스ID, 위치/시간, 코멘트).

# 9. 비기능 요구사항

## 성능/배터리
- 트래킹 기본 모드는 ‘균형(배터리-정확도)’로 설정하고, 필요 시 ‘정확도 우선’ 모드를 제공(설정).
- 정지 상태에서는 위치 업데이트 빈도를 줄여 불필요한 배터리 소모를 억제.
- 지도/POI 호출은 캐싱 및 호출 빈도 제한 정책 적용.

## 개인정보/보안
- 위치/이동 경로는 민감정보이므로 목적(산책 기록)과 보관 방식(로컬/서버)을 명확히 고지.
- 기록 삭제 기능 제공(전체 삭제 또는 개별 삭제).
- 서버 전송 시 HTTPS 적용, 접근 통제 및 최소 수집 원칙 준수.

# 10. 데이터/콘텐츠 설계
MVP에서는 코스 품질과 표준화를 우선한다. 초기 코스는 운영 가능한 수(예: 30–80개)로 시작하고, 업데이트 방식(앱 내 데이터 갱신 vs 앱 업데이트)을 결정한다.

## 핵심 엔터티(예시)
- **Route(코스)**: `route_id`, `name`, `districts`, `distance_m`, `est_duration_min`, `difficulty`, `tags`, `start/end` 좌표, `polyline`, `poi_summary`
- **POI(시설)**: `poi_id`, `category`, `name`, `lat/lng`, `source`, `meta(가능 시)`
- **WalkSession(기록)**: `session_id`, `route_id(optional)`, `start/end`, `duration`, `distance`, `track_polyline`, `stats`

# 11. 분석/측정(지표 및 이벤트)

## KPI
| KPI | 정의 | 목적 |
|---|---|---|
| Activation | 설치 후 24시간 내 코스 상세 진입 비율 | 초기 가치 전달 검증 |
| Start Rate | 코스 상세 대비 ‘산책 시작’ 클릭률 | 탐색→행동 전환 검증 |
| Completion Rate | 산책 시작 대비 ‘종료 저장’ 비율 | 트래킹 안정성/완주 경험 검증 |
| D7 Retention | 7일 재방문율 | 재사용성/습관화 가능성 |
| POI Usage | 산책 중 POI 카드 오픈률 | 주변 시설 정보 효용 검증 |

## 주요 이벤트(예시)
- `view_home`, `view_route_list`, `view_route_detail(route_id)`
- `click_start_walk(route_id)`, `walk_pause`, `walk_resume`, `walk_finish(session_id)`
- `poi_impression(category)`, `poi_click(poi_id, category)`
- `route_favorite_add/remove(route_id)`
- `permission_location_granted/denied`

# 12. 릴리즈 기준(Definition of Done)
- 테스트 코스 최소 5개에서 트래킹 안정성 검증(도심/공원 환경 포함).
- 위치 권한 거부/백그라운드 제한 상황에서의 UX(안내/대체 경로) 정리.
- 산책 기록 저장/조회 오류율 낮음 및 크래시 모니터링(베타 기간 포함).
- 코스 데이터 업데이트 절차 확정(앱 업데이트 없이 갱신할지 여부 포함).

# 13. 리스크와 대응
- GPS 정확도/배터리: 균형 모드 기본, 품질 낮음 배너, 노이즈 필터 적용.
- POI 품질: 카테고리 최소화(5종) + 반경/클러스터링 정책 + 제보 기능(후순위).
- 코스 운영 부담: 초기 코스 수 제한 + 템플릿 표준화(거리/시간/태그 기준).
- 권한 거부: 코스 탐색은 권한 없이도 가능, 트래킹 시작 시 재요청 및 이유 설명.
