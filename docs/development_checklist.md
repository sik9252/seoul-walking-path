# 서울걷길(관광지 수집형) 개발 체크리스트

기준: 관광지 방문 수집 게임형 MVP

## 0. 전환 정리
- [x] 산책/둘레길 중심 문서 제거 및 기획 전환
- [x] 앱 런타임에서 기존 산책 플로우 비활성화(App 셸 교체)
- [ ] 미사용 산책 코드/데이터 모듈 정리(2차 리팩터링)

## 1. 제품 정의
- [ ] 핵심 루프 확정: 탐험 -> 근접 판정 -> 카드 획득 -> 컬렉션 성장
- [ ] 방문 판정 반경 확정(예: 30m/50m/70m)
- [ ] 재수집 정책 확정: 동일 장소는 최초 1회만 수집(중복 수집 없음)
- [ ] 카드 등급/테마 규칙(랜드마크/자연/문화 등) 확정

## 2. 데이터 소스
- [x] 대한민국 관광지 공공 API 소스 확정(라이선스/호출 조건 문서화)
- [x] 지역 범위 1차 확정(서울 우선)
- [x] 관광지 원천 데이터 동기화 스크립트 구현(`backend/scripts/sync-tour-places.ts`)
- [ ] 좌표 누락 항목 보정 파이프라인 구현

## 3. 백엔드 도메인/API
- [x] 스키마 설계: `places`, `place_cards`, `user_place_visits`, `user_cards` (`backend/sql/tour_collection_schema.sql`)
- [x] API 구현: `GET /places`, `POST /visits/check`, `GET /cards/my`, `GET /cards/catalog`
- [x] 방문 판정 로직 구현(반경 기반)
- [x] 동일 장소 중복 수집 방지(최초 수집만 성공)

## 4. 프론트 구현
- [x] 탐험 탭: 현재 위치 기준 근처 관광지 목록(데모 좌표) + 페이지 추가 로드
- [x] 방문 성공/중복 수집 모달(Alert 기반) 연결
- [x] 카드 컬렉션 탭(획득 카드 목록) 연결
- [ ] 관광지 상세(설명, 이미지, 위치)

## 5. 운영/배포
- [ ] 백엔드 배포 플랫폼 확정(Render/Railway/Fly.io/Cloud Run)
- [ ] 프로덕션 DB(PostgreSQL) 연결
- [ ] 환경변수/시크릿 관리 체계 정리
- [ ] Android 내부 테스트 트랙 배포

## 메모
- 게임 정책: 동일 장소는 다시 방문해도 재수집되지 않음(쿨다운 없음, 1회 수집 고정).
- 초기 MVP는 서울 지역 관광지부터 시작하고, 이후 전국 확장.
- 현재 앱은 `EXPO_PUBLIC_API_BASE_URL(_ANDROID/_IOS)`를 통해 백엔드 `places/visits/cards` API를 호출.
- 관광 데이터/비용 기준 문서: `docs/tour_data_source_and_costs.md`
